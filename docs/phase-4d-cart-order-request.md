# Phase 4D Cart and Order Request

## What Was Added

Phase 4D adds a working V1 cart, order request checkout submission, protected
admin order viewing, and a hard-delete option for products.

Payment is still not collected on the website. Customers submit an order
request, and elara. confirms availability, payment, and delivery through the
customer's preferred contact method.

## Cart Behavior

- Cart state is stored in browser `localStorage`.
- Customers can add products from product cards and product detail pages.
- Cart items include product id, slug, name, image URL, unit price, quantity,
  and stock quantity when known.
- Customers can increase/decrease quantity, remove items, clear the cart, and
  proceed to checkout.
- Quantity cannot go below 1.
- If stock is known, quantity cannot exceed available stock.
- Out-of-stock products cannot be added to the cart.

## Checkout / Order Request Behavior

The checkout reads the cart from browser storage and submits:

- Full name
- Contact number
- Facebook profile link
- Instagram username
- Delivery address/location
- Preferred contact method
- Order notes
- Cart item details

On submit, the app creates:

- One `orders` row
- One `order_items` row per cart item

The order number is generated by the app and shown on the success page.

If Supabase is not configured, checkout shows:

```text
Order requests are not connected yet. Please contact us directly through Facebook or Instagram.
```

## Stock Handling

Stock is not deducted when an order request is submitted.

This is intentional because the customer has not paid yet and availability must
be confirmed manually. Stock deduction should happen in the next order workflow
phase when an admin marks an order as confirmed. If a confirmed order is later
cancelled, stock restoration can be handled in that later workflow.

## Migration 005

Run this migration manually in Supabase SQL Editor:

```text
supabase/migrations/005_order_request_submission_policies.sql
```

It adds:

- Public insert-only policies for `orders`
- Public insert-only policies for `order_items`
- Admin read policies for `orders`
- Admin read policies for `order_items`
- Admin update policy foundation for `orders`

It does not add public select, update, or delete policies for orders.

## Admin Orders Viewer

Protected admin routes were added:

- `/admin/orders`
- `/admin/orders/[orderId]`

The orders list shows order number, customer name, preferred contact method,
phone, social links, status, estimated total, created date, and item count.

The order detail page shows customer info, order notes, internal notes, ordered
items, total, and current status.

Status updates and stock deduction are intentionally deferred.

## Product Delete Behavior

Admin products now have a hard delete option in the product list and edit page.

Deactivate remains the safer default. If a product is already attached to an
order, deletion fails with:

```text
This product cannot be deleted because it is already attached to an order. You can deactivate it instead.
```

Storage files are not deleted in this phase.
